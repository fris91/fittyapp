Architectural Overview

This document presents the architectural diagrams of our web application designed to offer users functionalities such as AI-based diet creation, meal logging through barcode scanning or meal photos, and synchronization of physical activities from various fitness providers. The application also offers PRO features like consultations with nutritionists and personal trainers. The architecture ensures high scalability and resilience, utilizing microservices, centralized configuration, and service discovery.

Figure: The component diagram illustrates the main components of the application. The client app communicates with the API Gateway, which routes requests to the Subscription Service. The Subscription Service interacts with various microservices like User Service, Meal Service, Activity Service, and others. The Discovery Service (Eureka Server) and Config Server (Spring Cloud Config) provide centralized service discovery and configuration management. Databases like MongoDB and PostgreSQL store the application's data.

Figure: The sequence diagram shows the process of logging a meal through barcode scanning. The user interacts with the client app, which sends a request to the API Gateway. Authentication is verified via Keycloak. The request is then routed to the Subscription Service, which communicates with the Meal Service. The Meal Service retrieves food data from MongoDB and responds back through the chain to the client app.

Microservice Details

User Service

The User Service manages user information, authentication, and authorization. It interacts with the Discovery Service and Config Server for service discovery and configuration.

Subscription Service

The Subscription Service handles user subscriptions, orders, and plans. It acts as an intermediary between the client app and other microservices, providing a unified interface.
